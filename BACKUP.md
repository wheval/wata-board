# Backup & Restore Runbook

Wata-Board runs an automated PostgreSQL backup pipeline as a sidecar
container in `docker-compose.prod.yml`. This document covers the day-to-day
operation and the recovery path.

## Architecture

```
postgres ──┐
           │  pg_dump (every BACKUP_INTERVAL_SECONDS)
           ▼
       db-backup container ──► /backups volume ──► [optional] S3 off-site
                                       │
                                       └── .last-success marker
                                              │
                                              ▼
                                      backend /health/backup
```

- **What is backed up**: the `wataboard` Postgres database (all tables defined
  in `database/migrations/`).
- **What is NOT backed up**: Redis (intentionally — `tx-status` keys are
  ephemeral with a 24h TTL); SSL certs (regenerated by certbot); application
  logs (rotated by Winston; not authoritative state); env-var secrets (use a
  secrets manager).
- **Format**: `pg_dump --format=custom --compress=6` — restorable via
  `pg_restore`, parallelizable with `--jobs`.
- **Retention**: 7 daily, 4 weekly (Sunday promotions), 12 monthly (1st-of-month
  promotions). Override via `BACKUP_RETENTION_DAYS|WEEKS|MONTHS`.
- **Integrity**: every dump gets a `.sha256` sidecar, verified by both the
  existing `BackupVerifier` and `restore-postgres.sh`.
- **Encryption (optional)**: set `BACKUP_ENCRYPTION_PASSPHRASE` to enable GPG
  AES-256 symmetric encryption of every dump.
- **Off-site (optional)**: set `BACKUP_S3_BUCKET` to mirror to S3 (works with
  any S3-compatible store via `AWS_ENDPOINT_URL`).

## Required environment

Set in `.env` before `docker compose up`:

```bash
POSTGRES_DB=wataboard
POSTGRES_USER=wataboard
POSTGRES_PASSWORD=<strong-password>

# Optional — enables encryption
BACKUP_ENCRYPTION_PASSPHRASE=<long-random-string>

# Optional — enables off-site upload
BACKUP_S3_BUCKET=my-backups-bucket
BACKUP_S3_PREFIX=wataboard/postgres
AWS_ACCESS_KEY_ID=...
AWS_SECRET_ACCESS_KEY=...
AWS_DEFAULT_REGION=us-east-1
# AWS_ENDPOINT_URL=https://s3.example.com   # for non-AWS S3-compatible stores
```

> Store `BACKUP_ENCRYPTION_PASSPHRASE` somewhere other than the backup target.
> If the only copy lives in the same place as the encrypted backups, the
> encryption is decorative.

## Monitoring

- Backend exposes `GET /health/backup`:
  - `200 {status: "UP", lastSuccessAt, ageSeconds}` — fresh.
  - `200 {status: "UNKNOWN", lastSuccessAt: null}` — no marker yet (cold start).
  - `503 {status: "DOWN", lastSuccessAt, ageSeconds}` — stale (> 25h by
    default; tune via `BACKUP_MAX_STALENESS_SECONDS`).
- The `db-backup` container's own healthcheck flips to `unhealthy` on the
  same staleness threshold, so `docker ps` and the orchestrator both see it.
- Wire your alerting to the backend endpoint — it survives even if the backup
  container is the thing that's broken.

## Verifying a backup

```bash
docker compose -f docker-compose.prod.yml exec backend \
  npm run verify-backup
```

This invokes `BackupVerifier` ([backend/src/utils/backup-verifier.ts](backend/src/utils/backup-verifier.ts))
which checks file size, sha256, and SQL/dump markers on the most recent file
under `./backups`.

## Restoring

> **Restoring drops and recreates the schema.** It is destructive. Take the
> backend offline (or scale to 0) before running it against production.

1. Identify the dump to restore. Inside the backup container:

   ```bash
   docker compose -f docker-compose.prod.yml exec db-backup \
     ls -lh /backups/daily /backups/weekly /backups/monthly
   ```

2. (Optional) Pull from S3 if local copies are gone:

   ```bash
   docker compose -f docker-compose.prod.yml exec db-backup \
     aws s3 cp s3://$BACKUP_S3_BUCKET/$BACKUP_S3_PREFIX/daily/<file> /backups/daily/
   ```

3. Run the restore script:

   ```bash
   docker compose -f docker-compose.prod.yml exec db-backup \
     /scripts/restore-postgres.sh /backups/daily/wataboard-20260426T030000Z.dump
   ```

   For encrypted dumps (`.gpg`), pass `BACKUP_ENCRYPTION_PASSPHRASE` via the
   container env — it is already wired in compose.

4. Verify:

   ```bash
   docker compose -f docker-compose.prod.yml exec postgres \
     psql -U $POSTGRES_USER -d $POSTGRES_DB -c '\dt'
   ```

5. Bring the backend back up.

## Disaster recovery (lost host)

Assuming you have S3 off-site enabled:

1. Provision a new host, install Docker, clone the repo, populate `.env`.
2. `docker compose -f docker-compose.prod.yml up -d postgres db-backup`
3. Postgres comes up empty; the migrations in
   `/docker-entrypoint-initdb.d` run automatically. **Do not let this become
   the production state** — those are schema-only.
4. Pull the latest dump from S3 and restore (steps 2 and 3 above).
5. Start the rest of the stack.

Recovery time target with the defaults (daily backup, 1 GB DB,
S3 in same region): under 15 minutes.

## Testing the restore (recommended quarterly)

```bash
# Create a throwaway DB
docker compose -f docker-compose.prod.yml exec postgres \
  createdb -U $POSTGRES_USER restore_test

# Restore the latest daily into it
docker compose -f docker-compose.prod.yml exec db-backup \
  /scripts/restore-postgres.sh \
  /backups/daily/$(ls -t /backups/daily/*.dump | head -1) \
  --target-db restore_test

# Spot-check
docker compose -f docker-compose.prod.yml exec postgres \
  psql -U $POSTGRES_USER -d restore_test -c 'SELECT count(*) FROM users;'

# Clean up
docker compose -f docker-compose.prod.yml exec postgres \
  dropdb -U $POSTGRES_USER restore_test
```

A backup you have never restored is not a backup.
